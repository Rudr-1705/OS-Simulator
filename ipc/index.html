Inter-Process Communication

Normally, processes run in isolated memory spaces and cannot directly access each other's data. Inter-process communication (IPC) provides controlled ways for processes to exchange data, synchronize execution, and coordinate tasks. Explore how pipes, message queues, shared memory, and signals make that coordination possible.

Key Concepts

Pipe
Unidirectional byte stream between related processes. Anonymous, kernel-buffered. Created with pipe().
Message Queue
Allows passing typed, structured messages. Supports priority ordering. Persists until deleted.
Shared Memory
Fastest IPC — processes map same physical memory. Requires explicit synchronization (semaphores).
Signal
Asynchronous notification to a process. Very limited data (just the signal number). Used for control.
Blocking
Reader blocks when pipe empty; writer blocks when pipe full. Kernel suspends process until data available.
Named Pipe (FIFO)
Like pipe but has a filesystem path. Allows IPC between unrelated processes.

pipe()

IPC
int pipe(int pipefd[2]);

Pipes create a kernel-managed byte stream between a writer and a reader. The read end is pipefd[0], the write end is pipefd[1], and bytes move in FIFO order through a fixed-size buffer.

Parameters
pipefd[2] array where the kernel stores the two file descriptors, with index 0 for reading and 1 for writing.
System Calls Used
write(pipefd[1], buf, n); read(pipefd[0], buf, n);

The writer pushes bytes into the pipe and the reader removes them in the same order. If the pipe is empty, reads block. If it is full, writes block.

Typical Flow
close(pipefd[0]); close(pipefd[1]);

Processes close the ends they do not need so the kernel can detect EOF and broken-pipe conditions correctly.

Message Queues

Priority IPC
mqd_t mq_open(const char *name, int flags);

Message queues send complete messages instead of raw bytes. Each message can carry a priority, so the kernel can deliver urgent work before lower-priority items.

Parameters
name pathname-like identifier for the queue, such as "/jobs", so processes can open the same queue.
flags open behavior such as create, read-only, write-only, or non-blocking mode.
msg pointer to the message bytes being sent.
len size of the message buffer in bytes.
prio message priority used by the kernel to decide delivery order.
mqdes descriptor returned by mq_open() that identifies the queue for later send and receive operations.
Core Calls
int mq_send(mqd_t mqdes, const char *msg, size_t len, unsigned prio); ssize_t mq_receive(mqd_t mqdes, char *buf, size_t len, unsigned *prio);

mq_send() posts a full message with a priority, and mq_receive() dequeues the next available message from the kernel-managed queue.

Cleanup
mq_close(mqdes); mq_unlink("/queue-name");

mq_close() releases this process descriptor. mq_unlink() removes the named queue once it is no longer needed.

Shared Memory

Fastest IPC
int fd = shm_open("/region", O_CREAT | O_RDWR, 0666);

Shared memory maps the same physical pages into multiple processes, so communication happens through ordinary loads and stores rather than copying data through the kernel on every operation.

Parameters
"/region" shared-memory object name used by multiple processes to refer to the same region.
O_CREAT | O_RDWR flags that create the region if needed and open it for reading and writing.
0666 permission bits controlling which users can access the shared object.
size total bytes reserved for the region before mapping.
PROT_READ | PROT_WRITE protection flags describing how the mapped memory may be accessed.
MAP_SHARED tells the kernel that updates should be visible to every process mapping the same region.
Setup
ftruncate(fd, size); void *addr = mmap(NULL, size, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);

After mmap(), every process sees the same memory region. Reads and writes happen directly against the shared pages.

Why Synchronization Matters
sem_wait(&lock); access shared data sem_post(&lock);

The kernel does not automatically serialize access. Semaphores or mutexes are usually paired with shared memory to avoid races.

Signals

Asynchronous
int kill(pid_t pid, int sig);

Signals are lightweight asynchronous notifications. They are mainly used for control, interruption, and lifecycle events rather than transferring large amounts of data.

Parameters
pid target process ID, or a special value that can address process groups depending on the call.
sig signal number to deliver, such as SIGTERM, SIGSTOP, or SIGKILL.
act structure passed to sigaction() that defines the handler and signal behavior.
oldact optional structure where the previous handler configuration can be returned.
Core Calls
int sigaction(int sig, const struct sigaction *act, struct sigaction *oldact);

kill() sends the signal, and sigaction() lets a process install a handler for catchable signals.

Common Behavior
SIGTERM SIGKILL SIGSTOP SIGCONT

Signals are great for process control, but they carry very little payload compared with queues, pipes, or shared memory.
